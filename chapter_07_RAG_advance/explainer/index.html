1 What We Built

A complete Retrieval-Augmented Generation system that ingests 5,000 VWO A/B test cases and lets you ask questions or generate new test cases using hybrid search + AI.

🎯 The Challenge

QA teams at scale need fast access to thousands of test cases. Traditional search (keyword match) fails because:

  • Same concept expressed differently ("checkout flow" vs "purchase funnel")
  • Misspellings and abbreviations ("signup" vs "registration")
  • No semantic understanding — "P0 checkout bugs" doesn't match "critical payment failures"

💡 The Solution: RAG

Retrieval-Augmented Generation solves this by combining two powers:

🔍 Retrieval
Find relevant test cases semantically
🧠 Generation
Answer in natural language with citations

Our system uses a 5-stage pipeline that processes every query through: rewriting → embedding → hybrid search → re-ranking → generation.

🗄️ The Data: 5,000 VWO Test Cases

We generated 5,000 realistic VWO A/B testing scenarios covering 20 modules:

Landing Page Pricing Page Checkout Flow Signup Flow Navigation Homepage Hero Product Page Blog Layout Search Results Cart Page Payment Modal Onboarding Email Capture Mobile Menu Referral Program Live Chat

Each test case includes: jira_id, title, description, steps, expected, priority (P0–P3), module, and tags.

3 Embedding: bge-m3

One model, two types of vectors — the magic of BAAI's bge-m3.

🧠 What is an Embedding?

An embedding converts text into a list of numbers (a vector) that captures its meaning. Similar texts have similar vectors. "Checkout flow" and "purchase funnel" end up close together in vector space, even though they share no keywords.

Dense Vectors (1024 dimensions)

Every word contributes to a single, dense list of floating-point numbers. Captures overall semantic meaning. Example output:

[0.023, -0.145, 0.567, 0.001, -0.234, 0.789, -0.012, 0.345, ...]  (1024 values)

Sparse Vectors (Lexical Weights)

Only important tokens get non-zero weights — like TF-IDF but learned. Captures keyword precision. Example output:

{101: 0.45, 2057: 0.32, 89: 0.28, 3401: 0.21, 15: 0.18}  # token_id: weight

Dense — Semantic

# "test checkout flow"
→ [0.023, -0.145, ...]
# "test purchase funnel"
→ [0.025, -0.140, ...]  ← Close!

Sparse — Keyword

# "test checkout flow"
→ {"checkout": 0.5, "test": 0.3, "flow": 0.2}
# "test purchase funnel"
→ {"test": 0.3, "purchase": 0.4, "funnel": 0.3}

⚡ bge-m3: Best of Both Worlds

bge-m3 produces both dense and sparse vectors from a single forward pass through the model. This means we get semantic + keyword understanding with the same compute cost as a normal model.

# In Python:
model = BGEM3FlagModel("BAAI/bge-m3")
output = model.encode(["test checkout flow"],
                       return_dense=True,
                       return_sparse=True)
dense_vec = output["dense_vecs"]       # 1024-dim
sparse_vec = output["lexical_weights"]  # {token: weight}

The model is ~2.3 GB in FP16 and runs on CPU or GPU. We load it once as a singleton — the first query warms it up, subsequent calls are fast.

4 Hybrid Retrieval + RRF Fusion

The key innovation: combining dense and sparse search results.

🔍 The Problem

Dense search finds semantically similar results but can miss exact keyword matches. Sparse search is great at keyword matching but misses conceptually related content. Neither alone is perfect.

🤝 The Solution: Hybrid Search

We run both searches and merge results using Reciprocal Rank Fusion (RRF).

def rrf_fuse(dense_results, sparse_results, k=60):
    """Reciprocal Rank Fusion — merge two ranked lists."""
    scores = {}
    for rank, point in enumerate(dense_results):
        scores[point.id] += 1 / (k + rank + 1)
    for rank, point in enumerate(sparse_results):
        scores[point.id] += 1 / (k + rank + 1)
    # Sort by fused score descending
    return sorted(scores, key=scores.get, reverse=True)

🗄️ Qdrant — The Vector Database

We use Qdrant as our vector store. It natively supports both dense and sparse vectors in the same collection — a rare feature that makes hybrid search seamless.

# Collection config
client.create_collection(
    "vwo_test_cases",
    vectors_config=VectorParams(size=1024, distance="COSINE"),
    sparse_vectors_config=SparseVectorParams(
        index=SparseIndexParams(on_disk=True)
    )
)

Qdrant runs embedded (no Docker needed) — data is stored in a local folder ./data/qdrant_data/ using RocksDB. Zero infrastructure required.

5 Cross-Encoder Re-ranking

Fine-tuning retrieval quality with BAAI/bge-reranker-v2-m3.

📊 What is Re-ranking?

After hybrid retrieval gives us 20 candidate chunks, we need to pick the 4 best ones to send to the LLM. A re-ranker is a cross-encoder — it looks at the query and each candidate together and scores their relevance:

# Cross-encoder: (query, candidate) → relevance score
pairs = [
    ["What checkout tests exist?", "Title: Test guest checkout flow..."],  # score: 0.92
    ["What checkout tests exist?", "Title: Test pricing page layout..."],  # score: 0.31
    ...
]
scores = reranker.compute_score(pairs, normalize=True)

Unlike the bi-encoder (bge-m3) which embeds everything independently, the cross-encoder can use attention between query and candidate — much more accurate, but slower. That's why we only re-rank the top 20, not all 5,000.

Before Re-ranking (RRF)

#1: Pricing layout test    0.85
#2: Guest checkout test    0.72
#3: Signup flow test       0.68
#4: Checkout test          0.61
#5: Payment modal test     0.55

After Re-ranking

#1: Guest checkout test    0.94
#2: Checkout test          0.89
#3: Payment modal test     0.82
#4: Pricing layout test    0.31
#5: Signup flow test       0.25

Notice: the re-ranker correctly identified that "guest checkout" is most relevant to the query about checkout tests, boosting it from #2 to #1. The pricing test dropped from #1 to #4 because it's less relevant.

"Create a test case for VWO-9999"

→ **Title:** Test email capture popup timing
**Preconditions:** Exit-intent detection...
**Steps:**
1. Load landing page...
2. Move cursor toward address bar...
**Expected:** Popup triggers after 3s
**Priority:** P2
**Tags:** Conversion, A/B Test

Detection is keyword-based for speed: if the query contains "create", "generate", "new test case", or "jira", it routes to Generate mode.

🔄 Query Rewriting

Before searching, we use a cheap LLM (GPT-4o-mini via Openrouter) to generate 3 alternative phrasings of the user's query. This increases recall by covering different ways to express the same information need.

User query: "checkout bugs"
→ "checkout flow defects"
→ "payment issues in checkout"
→ "errors during purchase process"

All 4 queries are embedded and searched — results merge via RRF.

Upload & Preview

Accepts CSV or Excel files. Shows row count, column types, first 5 rows. Lets you pick which columns are text (for embedding) vs metadata (for filtering).

Live Pipeline Tracking (SSE)

Server-Sent Events stream real-time updates as the pipeline runs. Each stage shows detail cards with histograms, vector previews, and progress bars. No page refreshes needed.

Paginated Chunk Viewer

Browse 50 chunks per page with substring search and filters by priority/module. Each chunk shows its text, metadata badges, dense vector preview, and sparse token weights.

Interactive Q&A

Left pane shows live pipeline stage progress. Right pane is the chat. After each query, expandable sections show the rewrites, search comparison (dense vs sparse vs RRF), re-ranking before/after table, and the final answer with [Chunk N] citations.
