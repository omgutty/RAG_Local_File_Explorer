What We Built

A production-grade RAG system that ingests test cases and lets you ask questions or generate new ones using hybrid AI search.

🎯 The Problem

QA teams at scale need instant access to thousands of test cases. Traditional keyword search fails because:

  • Vocabulary mismatch — "checkout flow" vs "purchase funnel" share no keywords but mean the same thing
  • No semantic understanding — "P0 checkout bugs" won't match "critical payment failures"
  • Context loss — searching through 5,000 rows in a CSV or Confluence page is impractical

💡 The Solution: RAG

Retrieval-Augmented Generation combines two superpowers:

🔍 Semantic Retrieval
Find relevant chunks by meaning, not just keywords
🧠 LLM Generation
Answer in natural language with chunk citations
📊 Hybrid Search
Dense + sparse vectors fused via RRF
🎯 Cross-Encoder
Re-rank candidates for maximum precision

Flask receives the uploaded CSV/XLSX, saves it with a UUID filename, and pandas reads it. The UI shows a preview table with 5 sample rows, column names, and data types. User selects which columns are text (for embedding) vs metadata (stored as payload for filtering).

df = pd.read_csv("data/uploads/abc123.csv")  # 5000 rows × 8 columns
text_cols = ["title", "description", "steps"]  # concatenated per row
meta_cols = ["jira_id", "priority", "module"]   # stored as payload

Each row's text columns are concatenated into a single document, then split into overlapping chunks using recursive character splitting. The splitter tries boundaries in order: \n\n → \n → . → , → space.

# config.py
CHUNK_SIZE    = 1000   # Max chars per chunk
CHUNK_OVERLAP = 150    # Overlap between adjacent chunks

# One document (a single test case) → multiple chunks
"Title: Test guest checkout flow...\n\nDescription: Verify..."
  → Chunk 0: "Title: Test guest checkout flow...\n\nDescription: Verif"
  → Chunk 1: "fy the guest user can complete purchase without account...\n\nSteps:"
                   ↑ 150 chars of overlap ↑

Each chunk is passed through bge-m3 which produces two vector representations from a single forward pass:

Dense Vector (1024-dim)
[0.023, -0.145, 0.567, 0.001, -0.234, ...]  # 1024 floats
← Captures semantic meaning
Sparse Vector
{101: 0.45, 2057: 0.32, 89: 0.28, ...}
← Captures keyword importance

Both vectors and metadata are upserted into Qdrant (embedded mode — no Docker needed). The collection is configured with dense (1024-dim, COSINE) and sparse vector support.

# Qdrant collection config
client.create_collection("vwo_test_cases",
    vectors_config=VectorParams(size=1024, distance="COSINE"),
    sparse_vectors_config={"sparse": SparseVectorParams(...)}
)

# Each point stores:
PointStruct(
    id=0,                                                    # sequential
    vector={"": dense_vec, "sparse": {indices, values}},
    payload={"text": "...", "priority": "P0", "module": "Checkout"}
)

Points are upserted in batches of 100. Qdrant persists to data/qdrant_data/ using RocksDB.

Embedding: bge-m3

One model that produces two vector types — the secret sauce behind our hybrid search.

🧠 What is an Embedding?

An embedding converts text into a list of numbers (a vector) that captures its meaning. Similar texts end up with similar vectors — "checkout flow" and "purchase funnel" are close in vector space even though they use different words.

📐 Dense Vectors (1024 Dimensions)

Every word in the text contributes to a dense, continuous vector. This captures semantic similarity — the overall meaning.

"Test guest checkout flow with payment modal"
  → [0.023, -0.145, 0.567, 0.001, -0.234, 0.789, ...]  # 1024 floats
       ↑                                 ↑          ↑
      Each dimension = a learned semantic feature

🔤 Sparse Vectors (Lexical Weights)

Only important tokens get non-zero weights. This captures keyword-level precision — exact term matching, just with learned importance scores instead of raw TF-IDF.

"Test guest checkout flow with payment modal"
  → {
      "test":    0.32,    # important but common
      "guest":   0.45,    # distinctive keyword
      "checkout":0.52,    # highly relevant
      "payment": 0.38,    # also relevant
      "modal":   0.28     # specific term
    }
  ← Only ~10-30 tokens get non-zero weights

⚡ Why bge-m3?

BAAI/bge-m3 (Beijing Academy of AI) produces both dense and sparse vectors from a single forward pass. This means we get semantic + keyword understanding at the same computational cost as a normal model.

Hybrid Retrieval + RRF Fusion

The core innovation: combining dense semantic search with sparse keyword search for the best of both worlds.

🤔 The Problem

Dense search finds semantically similar results but can miss exact keyword matches. Sparse search is great at keyword matching but misses conceptually related content. Neither alone is good enough.

🔍 Dense Search Only

Finds "purchase funnel" when searching "checkout" — but might miss test cases that use the exact keyword "payment" (which a user explicitly searched for).

📖 Sparse Search Only

Finds exact "checkout" matches but misses test cases described as "purchase funnel" or "buy flow" — even though they're semantically identical.

🤝 Reciprocal Rank Fusion (RRF)

We run both searches and merge results using RRF — a simple but powerful algorithm:

def rrf_fuse(dense_results, sparse_results, k=60):
    scores = {}
    for rank, point in enumerate(dense_results):
        scores[point.id] += 1 / (k + rank + 1)
    for rank, point in enumerate(sparse_results):
        scores[point.id] += 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

# k=60 controls how much high ranks matter
# Items that rank well in BOTH lists get boosted

Cross-Encoder Re-ranking

Fine-tuning retrieval precision with BAAI/bge-reranker-v2-m3 — the final quality gate before the LLM.

📊 Bi-Encoder vs Cross-Encoder

Understanding the difference is key to understanding why we need both:

🧠 Bi-Encoder (bge-m3)

Encodes query and document separately into vectors. Fast — can pre-compute all document vectors. Used for retrieval (narrow 5,000 → 20).

query_vec = model.encode("checkout test")
doc_vec   = model.encode("Test guest checkout")
# Cosine(query_vec, doc_vec) = 0.87
🎯 Cross-Encoder (reranker)

Processes query and document together through attention. Slower but much more accurate. Used for re-ranking (narrow 20 → 4).

pairs = [["checkout test", "Test guest checkout"]]
score = reranker.compute_score(pairs)
# Direct relevance score = 0.94

The cross-encoder uses attention between query and candidate, so it can reason about their relationship directly. This is why it's more accurate — but also why we only apply it to the top 20 candidates (not all 5,000).

LLM Generation with Openrouter

Converting retrieved chunks into grounded answers or structured test cases using DeepSeek via Openrouter API.

🎯 Two Modes: Answer vs Generate

The system auto-detects user intent based on keywords. If the query contains "create", "generate", "new test case", or "jira" — it routes to Generate mode. Otherwise, it answers with citations.

System: We have 8 checkout-related test cases covering payment failures:

- [VWO-120] Test credit card decline during checkout — verifies the error message appears and user can retry with a different card.
- [VWO-305] Test PayPal timeout during payment — verifies the modal shows a timeout message and user can switch to credit card.
- [VWO-521] Test network failure at payment confirmation — verifies the order is saved as pending and user sees a "try again" prompt.

🔄 Query Rewriting (Boosting Recall)

Before searching, a cheap LLM (GPT-4o-mini, $0.15/1M tokens) generates 3 alternative phrasings of the user's query. All 4 queries (original + 3 rewrites) are embedded and searched, then results are merged via RRF.

# User types:
"checkout bugs"

# GPT-4o-mini rewrites to:
1. "checkout flow defects"
2. "payment issues during checkout"
3. "errors in purchase process"

# All 4 queries are searched → results fused → best chunks win

This dramatically increases recall by covering different phrasings of the same intent.

📱 Four Core Pages

The app has a sidebar navigation with 4 main sections. All data flows through Server-Sent Events (SSE) — no polling, no page refreshes.

Upload & Preview — Drop a CSV or Excel file. The app reads it with pandas and shows a preview table (first 5 rows, data types, column names). You select which columns are text (for embedding) and which are metadata (stored as Qdrant payload).

Live Pipeline Tracking — Server-Sent Events stream real-time updates as the ingest pipeline runs. Each stage shows detail cards: chunk length histograms, vector previews (first 8 dense dimensions, top 5 sparse tokens), and batch progress bars.

Paginated Chunk Browser — Browse 50 chunks per page with substring search and dropdown filters for priority (P0–P3) and module. Each chunk card shows text, metadata badges, a vector preview, and cited references from the last chat.

Interactive Q&A — Left pane shows pipeline stage progress (rewrite → embed → search → rerank → generate). Right pane is the chat. After each query, expandable sections show rewrites, search comparisons (dense vs sparse vs RRF), re-ranking before/after tables, and the final answer with [Chunk N] citations.
